# Phase 0 Research: Restaurant Demo App

User input for `/speckit-plan` named the major stack picks. This file
records the smaller decisions and rejected alternatives.

## Decisions

### D1 — SQLite over Postgres

**Decision**: SQLite via Prisma, file-based at
`restaurant-app/server/prisma/dev.db`.

**Rationale**: Constitution Principle IV ("Local-First Runnability"). The
transcript uses Neon Postgres which requires the user to provision a DB and
paste a connection string. SQLite removes that step; Prisma's API is
identical between the two.

**Alternatives considered**:

- **Neon Postgres** (transcript's choice) — REJECTED: external setup.
- **PG via Docker** — REJECTED: requires Docker, more friction than the demo
  earns.
- **In-memory JSON store** — REJECTED: doesn't survive a restart, can't
  demonstrate Prisma's value.

### D2 — JavaScript, not TypeScript

**Decision**: Plain JS for both client and server.

**Rationale**: Constitution Principle I. TypeScript adds a build step on the
backend (or `ts-node`) and significant config on the frontend (`tsconfig`,
type imports). The demo's audit trail value is "look at the spec → look at
the code" — no type generics needed for that to land.

**Alternatives considered**:

- **TypeScript everywhere** — REJECTED: adds 4–6 config files and learning
  cost without changing what the demo demonstrates.
- **TS on server, JS on client** — REJECTED: split-brain mental model.

### D3 — Vite + React (not Next.js / CRA)

**Decision**: Vite + React 18 with `@vitejs/plugin-react`.

**Rationale**: Vite is the current standard for new React SPAs (CRA is
deprecated). Next.js bundles SSR and routing we don't need; SPA + REST API
is the cleanest separation of concerns to show in a demo.

**Alternatives considered**:

- **Next.js** — REJECTED: adds App Router/Pages Router decision, server
  components vs client components, more concepts than needed.
- **Create React App** — REJECTED: deprecated.
- **Plain HTML + React via CDN** — REJECTED: Tailwind via PostCSS doesn't
  work without a build step.

### D4 — JWT in `localStorage`

**Decision**: Store the JWT in `localStorage`. Send as
`Authorization: Bearer <token>` on every API call. No refresh tokens.

**Rationale**: Demo scope. Production apps would prefer httpOnly cookies +
refresh tokens to reduce XSS surface, but that requires CSRF protection and
two endpoints, neither of which the demo's spec requires. The transcript
also uses an access-only JWT pattern.

**Alternatives considered**:

- **HttpOnly cookies + CSRF** — REJECTED: out of scope for a demo.
- **Refresh tokens** — REJECTED: same.
- **Session in DB** — REJECTED: extra table, extra middleware, no demo
  value.

### D5 — `bcryptjs`, not native `bcrypt`

**Decision**: `bcryptjs` (pure JS).

**Rationale**: Native `bcrypt` requires a C compiler at install time. On
fresh dev environments (especially WSL/Windows), this is a frequent failure
mode. `bcryptjs` is slower in absolute terms but indistinguishable at demo
scale (≤dozens of users).

**Alternatives considered**:

- **`bcrypt`** — REJECTED: install fragility.
- **`argon2`** — REJECTED: same compilation issue, no measurable security
  benefit for the demo.

### D6 — Tailwind via PostCSS

**Decision**: Standard Tailwind setup (`tailwind.config.js`,
`postcss.config.js`, directives in `src/index.css`).

**Rationale**: Vite has first-class support; one of the most common React
patterns. Avoids the "Tailwind via CDN" gotcha (no JIT, no purge — entire
~3MB CSS file shipped).

**Alternatives considered**:

- **Tailwind CDN** — REJECTED: ships unused CSS, no design tokens.
- **CSS Modules / vanilla CSS** — REJECTED: uniform with the transcript's
  pick; Tailwind keeps us out of the styling weeds.

### D7 — One JWT secret, not key rotation

**Decision**: Single `JWT_SECRET` from `.env`, default value generated by
seed script if missing.

**Rationale**: Demo. Production would rotate via a KMS.

### D8 — Seed strategy: idempotent upsert

**Decision**: `seed.js` uses `upsert` keyed on natural keys (category name,
dish name + category) so re-running it is safe.

**Rationale**: Lets the user re-run `npm run db:setup` after schema changes
without resetting the DB.

**Alternatives considered**:

- **`prisma migrate reset` then create** — REJECTED: loses any local data
  the user added during exploration.

### D9 — Image hosting

**Decision**: External HTTPS URLs (Unsplash) referenced from seed data.
Frontend `<img onError>` falls back to a placeholder data URL.

**Rationale**: Avoids bundling binary assets. Unsplash hotlinking is allowed
under their license for demos.
